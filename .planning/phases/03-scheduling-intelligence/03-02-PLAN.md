    **Method 1: `async checkConflicts(params)`**
    ```typescript
    async checkConflicts(params: {
      start: EventTime;
      end: EventTime;
      calendarUrls?: string[];  // optional filter; default = all calendars across all accounts
      accountId?: string;       // optional account filter
    }): Promise<ConflictResult>
    ```

    Implementation:
    1. Convert proposed start/end to epoch ms via eventTimeToMs
    2. Compute fetch window: use proposed start minus 1 year as fetch start (to catch recurring masters whose DTSTART is before proposed range), proposed end as fetch end. Per RESEARCH.md Open Question 3, use wider window to catch recurring masters.
    3. Determine which calendars to check:
       - If calendarUrls provided, fetch from those specific calendars (resolve account for each)
       - If accountId provided, fetch all calendars for that account
       - Otherwise, fetch all calendars across all connected accounts (use _resolveClients())
    4. For each calendar: fetchCalendarObjects with time range, collect raw ICS data strings (obj.data)
    5. Call expandToBusyPeriods(allICSObjects, proposedStartMs, proposedEndMs) -- but use wider window (fetch start to fetch end) for expansion too so recurring instances are correctly captured
    6. Actually: expand with the wider window but detect conflicts against the proposed time only. So: expandToBusyPeriods(allICS, wideWindowStartMs, wideWindowEndMs), then mergePeriods(), then detectConflicts(proposedStartMs, proposedEndMs, merged)
    7. Return ConflictResult with hasConflict, conflicts array, and no suggestions (caller uses suggestSlots for that)

    **Method 2: `async suggestSlots(params)`**
    ```typescript
    async suggestSlots(params: {
      durationMinutes: number;
      searchStart: EventTime;       // start of search window
      searchDays?: number;          // default 7, per locked decision
      calendarUrls?: string[];
      accountId?: string;
      workingHoursStart?: number;   // 0-23, optional
      workingHoursEnd?: number;     // 0-23, optional
      maxSlots?: number;            // default 5, per locked decision
    }): Promise<SlotSuggestion[]>
    ```

    Implementation:
    1. Compute search window: searchStartMs from eventTimeToMs, searchEndMs = searchStartMs + searchDays * 86400000
    2. Fetch all calendar objects in window (same calendar resolution logic as checkConflicts, using wider fetch window)
    3. expandToBusyPeriods, mergePeriods
    4. Call findAvailableSlots with durationMs = durationMinutes * 60000, working hours params, maxSlots (default 5), slotTzid = searchStart.tzid
    5. Return SlotSuggestion[] (caller converts ms to EventTime for display)

    **Helper: `private async _fetchAllICS(params)`**
    Extract the common calendar-fetching logic (resolve calendars, fetch objects, collect data strings) into a private helper to avoid duplication between checkConflicts and suggestSlots.
